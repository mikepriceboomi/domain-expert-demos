What this demo does

When Sarah Chen's record appears in Workday, one Boomi process sees it and kicks off everything else. No one hits a button. No one opens a ticket. The orchestrator takes that single event and fans out into five parallel workstreams — identity, collaboration, IT setup, compliance, and day-one logistics — all running simultaneously.

By the time the demo ends, six systems are provisioned, a compliance gate has been cleared, and Sarah has a full day-one calendar in her inbox. The whole thing is built from one master process and five subprocesses, all in the Boomi platform.

The integration stat
The Companion-built suite contains 6 processes, 6 connections, 17 operations, 11 JSON profiles, 10 maps, and 1 cross-reference table. All deployed to a single Boomi atom in the Domain-Experts environment.

📖
The story

The fictional company is Meridian. Sarah Chen is starting her first day as Business Development Manager. She sat down, opened her laptop, and everything worked.

In most companies, that outcome would require someone in IT to have opened tickets days earlier — one for Okta, one for M365, one for Slack, a ServiceNow ticket for the hardware request. Some would be done by day one. Some wouldn't.

At Meridian, the moment Sarah's record hit Workday, Boomi saw it. Six systems provisioned before she arrived. Zero tickets raised.

Demo new-hire record
The demo uses a built-in test record: Priya Raman, Sr. Solutions Consultant, Pre-Sales Engineering, manager David Whitmore, start date 2026-07-06. This fires automatically when you run the master orchestrator — no manual trigger needed.

How it's wired

The architecture is a hub-and-spoke: one master orchestrator calls five subprocesses in sequence. Each subprocess owns one domain of the onboarding lifecycle and is independently maintainable.

Workday (new employee record)
          │
          ▼
Master Orchestrator
          │
  ┌──────┬───┴───┬──────┬──────────┐
  ▼      ▼       ▼      ▼          ▼
Okta    M365/Slack ServiceNow Policy   Day One
        Zoom               Gate    Schedule

State flows through the entire chain via Dynamic Process Properties (DPPs) — think of these as variables that persist across every subprocess call. The Okta user ID created in step 1, for example, is automatically available to every process that comes after it.

Key state variables

PropertyWhat it carries
DPP_EMPLOYEE_EMAILWork email — used by Collaboration, Policy gate, and Day One
DPP_OKTA_USER_IDCreated in step 1, flows into every later activation and app assignment
DPP_ONBOARDING_STATUSGate variable — master orchestrator checks this before moving to Policy
DPP_DOCUSIGN_ENVELOPE_IDTracks the policy envelope through the polling loop
DPP_SNOW_SYS_IDServiceNow user ID — links the onboarding ticket to the right person
DPP_EMPLOYEE_START_DATEUsed by Day One to calculate all four calendar event timestamps
Why this matters in the demo
State threading is one of the strongest technical points to make. The Okta user ID that gets created in subprocess 1 is automatically available to subprocess 2, 4, and 5 — with no re-lookup, no extra API call, no manual handoff. That's the orchestration layer doing its job.

🔐
Subprocess 1 — Identity & Access

Okta

This subprocess creates Sarah's identity in Okta and assigns her to the right access groups based on her department. It uses a cross-reference table to map department name to Okta group — so a Business Development hire automatically gets the BD-Apps-Group, a Pre-Sales hire gets SE-Apps-Group, and so on. No manual lookup required.

1
Create Okta user — staged (not yet active), splits full name into first/last automatically
2
Extract Okta user ID — written to DPP_OKTA_USER_ID immediately, so every downstream process has it
3
Assign to app group — cross-reference table maps department → group automatically
4
Activate user — transitions from STAGED to ACTIVE

If anything fails, the subprocess writes identity_failed to the onboarding status property and returns — the master orchestrator catches this at the decision gate and alerts the HR team.

Subprocess 2 — Collaboration Tools

M365 Slack Zoom

This is the most visually impressive subprocess on the canvas — it uses a Branch shape to run three provisioning tracks in parallel. M365, Slack, and Zoom all go simultaneously. This is the moment to make in the demo: what used to be three separate tickets running sequentially is now one Branch shape firing all three at once.

A
Microsoft 365 lane — create user, assign E3 licence, provision OneDrive, add to Teams department channel
B
Slack lane — create user in workspace, add to #general, #announcements, and department channel
C
Zoom lane — create user with Pro licence

All three status results are consolidated at the end into a single provisioning summary before returning to the orchestrator. M365 and Teams both run through the same Microsoft Graph connection — Teams is just a different Graph API endpoint, not a separate system.

🎫
Subprocess 3 — IT Service Desk Setup

ServiceNow

Creates Sarah's user record in ServiceNow and automatically raises an onboarding ticket assigned to the IT Onboarding team. The ticket due date is set to Sarah's start date, with priority 3 (Moderate). This is the ticket that would normally be raised manually — here it's created the moment the orchestrator runs.

1
Create ServiceNow user record — name, email, department, manager, user type = employee
2
Extract sys_id — the ServiceNow user ID, written to DPP_SNOW_SYS_ID
3
Create onboarding ticket — linked to the user via sys_id, due date = start date, assigned to IT Onboarding
Real error handling here
Unlike some other subprocesses, the ServiceNow calls use actual try/catch — a non-2xx response genuinely throws an error rather than being simulated. This one is wired closer to production behaviour.

📋
Subprocess 4 — Policy Gate

DocuSign Okta M365

This is the governance moment of the demo. Nothing goes fully active until Sarah signs the Acceptable Use Policy. The subprocess sends the "Meridian Acceptable Use Policy v4.2" DocuSign envelope, then enters a polling loop — checking status every 5 minutes, for up to 24 hours. If she doesn't sign, it sends a reminder and keeps checking.

The moment the envelope comes back as completed, Okta app assignments flip from PROVISIONED to ACTIVE, and M365 sign-in is enabled. The canvas shows the polling loop as a visible back-edge — it's one of the most interesting shapes in the whole build.

1
Send DocuSign envelope — AUP v4.2 template sent to Sarah's work email
2
Poll for signature — checks status, increments a counter, loops back every 5 minutes
3
24-hour reminder — if 288 polls hit without a signature, sends an M365 reminder email and resets the counter
4
On completion — activates Okta app assignments and enables M365 sign-in

M365 Calendar M365 Mail

Live vs. mock

The demo runs cleanly on placeholder credentials — no live systems needed. Here's what's real and what's simulated, and what it takes to make each thing live.

What Status To make it live
ServiceNow error handling LIVE Already using real try/catch — non-2xx genuinely throws
Connections (Okta, M365, Slack, Zoom, DocuSign) MOCK Add real credentials via Boomi environment extensions
HTTP 201 success checks MOCK Replace simulated decision with real response-code check or try/catch
DocuSign envelope completion MOCK Extract real status from the Get-Envelope API response
5-minute poll cadence MOCK Enforce via scheduled re-run (Boomi has no native sleep)
Notification email recipients MOCK Set real addresses on the mail operation or via environment extensions
If someone asks "is this real?"
Yes — the processes are real Boomi integrations running on a live atom. The connector credentials are demo placeholders, but the architecture, the orchestration logic, the parallel branching, the polling loop, the state threading — all of it is genuinely built and deployed. This suite was built by Boomi Companion in a single session. That's part of the story too.
