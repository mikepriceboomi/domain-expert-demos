Ch 01 Sources

Every enterprise runs on fragmented data. Thousands of sources. Each with its own credentials, its own endpoints, its own quirks. Before you can analyze data, move it, or run AI on top of it — you have to be able to get to it.

There are four categories that matter here.

APIs. Systems like Salesforce, NetSuite, Jira, and marketing platforms. Each one behaves differently, and a typical enterprise runs dozens of them.

Databases. SQL Server, Oracle, MySQL, PostgreSQL. These are the operational core — often running on-prem, often the most sensitive, and the hardest to replicate without disrupting the business.

SAP. A category on its own, and for good reason. You'll see why shortly.

Files. CSV, SFTP, FTP, and managed file transfer at scale.

This is the starting point: fragmented sources, and the need to do something intelligent with what's inside them.

Ch 02 Boomi Data Integration

BDI — Boomi Data Integration — is how we bridge from fragmented sources to everything downstream.

Think of it in two halves.

The left side is ingestion: reading data from your sources and getting it into your environment. There are five connector types on the ingestion side, each designed for a different class of source.

The right side is transformation and orchestration: shaping that data and coordinating what happens to it once it's there.

We'll walk through all five next.

Ch 03 Connector Types

Native connectors. Around two hundred of them, covering the most common API-based sources. When a vendor changes an endpoint, updates pagination, rotates credentials — Boomi maintains the connector. The data flow keeps running without intervention.

CDC — Change Data Capture. The most efficient way to get data out of operational databases. Instead of querying the database directly, CDC reads from the transaction logs. No load on the production system. No impact on the web and mobile applications running on top of it. For customers whose databases are load-bearing in production, this is the only acceptable approach.

SAP Data Connector. Built natively into BDI, and paired with Boomi for SAP — which runs on-prem inside the SAP environment. Every table, every dataset, every trigger in SAP can be streamed through Boomi's event layer into BDI. It's effectively CDC for SAP. Very few vendors can offer this, and it's why SAP is its own category.

DCA — Data Connector Agent. Not every source has a native connector, and building one for every system in the world isn't scalable. DCA solves this with AI. Point it at any third-party system — say, GitHub — and it reads the vendor documentation, identifies the endpoints, credentials, and pagination patterns, and generates a working connector on the fly. This unlocks thousands of systems that would otherwise require custom development.

MFT — Managed File Transfer. Native connectors for FTP and SFTP, plus governed, secured file transfer at scale. MFT currently runs as a standalone console, with native BDI integration on the roadmap.

Ch 04 Knowledge Hub

To run AI agents, you need more than structured data. You need context.

That context lives in unstructured sources — documents, PDFs, images, location feeds. Traditional data warehouses weren't built for this. The Knowledge Hub was.

It's a native extension of the data integration layer, not a separate product bolted on. Any source you can connect to in BDI, you can feed directly into the Knowledge Hub — where it's vectorized and made available for semantic retrieval.

The result is a context layer your agents can trust — continuously updated, governed at the source, and accessible without custom pipelines.

Ch 05 Data Hub

Data Hub is the golden record layer. Data stewardship, deduplication, and unified master data — all in one place.

There's a field conversation worth having here. When you're implementing integrations across three or more systems, Data Hub belongs in the design. Point-to-point integrations between six or seven systems become an architectural tangle — hard to govern, expensive to change, impossible to scale. Data Hub in the middle keeps the architecture clean. Every system connects to one hub. Adding new systems becomes additive, not exponential.

A native integration between BDI and Data Hub makes this significantly more powerful in two directions.

Data Hub as a source: customers want to analyze their golden records in their analytics environment and run agents on top of them. That native integration makes it seamless — golden record datasets flow directly into any data or AI service downstream.

Data Hub as a target: onboarding new sources into Data Hub should be fast. Point any BDI source at Data Hub in a few clicks — less configuration, faster time to value.

Together, these capabilities turn Data Hub from a master data store into an active participant in the AI data architecture.

Ch 06 Retailer Use Case

Here's what this looks like end-to-end.

A large retailer. Complex operations, multiple HR systems from years of acquisitions, logistics spanning dozens of distribution locations. They want to build an AI agent to optimize their distribution network — routes, schedules, driver assignments.

Here's how it works.

BDI ingests from every relevant source. Salesforce provides CRM and marketing data — via native connector. Operational databases carry inventory and purchase history — via CDC, with zero impact on production. SAP handles logistics and finance — via the SAP Data Connector. Photos, PDFs, and location data come in as files — via MFT.

From there, the data splits by type.

Unstructured data — PDFs, location feeds, photos — flows into the Knowledge Hub, where it's vectorized and made available for semantic retrieval.

Structured records — employee data, driver data — flow into Data Hub, where they're deduplicated and merged into clean golden records. The agent knows who every driver is, across every system that's ever touched that record.

The agent runs in Agent Studio. It draws on the Knowledge Hub for context, and on the Data Hub golden records for operational ground truth. The result is an agent that can answer real operational questions — not from stale exports or disconnected reports, but from a live, governed, unified data foundation built on every source the business runs on.

That's the data activation story: get the data, improve the data, activate it for AI.
